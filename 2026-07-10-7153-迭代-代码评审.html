🔴 阻断项（必须修改，-42 分）
阻断 接口权限粒度过粗，非商品管理角色无法查看商品弹窗 GoodsBaseController.java:523 安全
问题分类：安全

queryGoodsInfoByCommonUi 方法使用了 @RequiresPermissions("goods:goodsBase:view") 注解。

该接口是一个通用商品信息弹窗查询接口，调用方为 goodsBasicDetail 指令，已嵌入到：

  • 内部订货单 — invOrderGoodsDetailInnerList.jsp 商品名称列
  • 批发销售单 — invWholesaleSalesDetailList.jsp 商品名称列

如果用户有内部订货单/批发销售单的权限，但没有商品资料管理权限（goods:goodsBase:view），点击商品名称弹窗时接口返回 403，弹窗无法展示。

建议：去除 @RequiresPermissions("goods:goodsBase:view") 注解。该接口仅查询商品基本信息（名称、条码、图片等），不涉及编辑/删除，属于只读通用接口，无需商品资料管理权限。
阻断 NPE 风险：查询结果未做 null 判断 GoodsBaseController.java:528-529 健壮性
问题分类：健壮性

goodsBaseService.getGoodsBasicInfo(goodsId) 可能返回 null（商品不存在或 goodsId 为无效值），紧接着 goodsDetailValue.setImage(...) 会抛出 NullPointerException。

同文件中的 getGoodsDetail 方法（第 542 行）对此做了正确的 null 判断。

public Object queryGoodsInfoByCommonUi(@RequestBody String goodsId) { GoodsDetailValue goodsDetailValue = goodsBaseService.getGoodsBasicInfo(goodsId); goodsDetailValue.setImage(InvUtils.billImageOssUrl(goodsDetailValue.getImage())); // ← NPE if null Map<String, Object> map = new HashMap<>(); map.put("goodsInfo", goodsDetailValue); return map; } // 对比现有 getGoodsDetail 方法（第 542 行）—— 有 null 保护： GoodsDetailValue goodsDetailValue = goodsBaseService.getGoodsDetailValue(cond); if (goodsDetailValue != null) { // ← 正确的做法 // ... 业务处理 }
建议：在操作 goodsDetailValue 之前添加 null 检查：if (goodsDetailValue == null) { return null; }

建议 注释错位：Javadoc 悬空在新方法上方 GoodsBaseController.java:514-518 注释规范
问题分类：注释规范

第 514-517 行的 Javadoc（"根据商品信息id查询商品信息 用于查询和编辑"）原本属于 getGoodsDetail 方法（第 535 行）。新方法插入后，该注释被悬空在 queryGoodsInfoByCommonUi 上方，而 getGoodsDetail 方法失去了自己的注释，容易误导维护者。

/** * 根据商品信息id查询商品信息 用于查询和编辑 ← 属于 getGoodsDetail，放错位置了 */ /** 商品基本信息弹窗（轻量查询，仅查两张表） */ public Object queryGoodsInfoByCommonUi(...) { ... } ← getGoodsDetail 现在没有注释了 public Object getGoodsDetail(...) { ... }
建议：删除第 514-517 行悬空的旧注释。
建议 Controller 入参缺少基础校验 GoodsBaseController.java:526 健壮性
问题分类：健壮性

@RequestBody String goodsId 直接接收 POST body 作为 goodsId，未校验 null/空字符串。前端传入空 body 时会执行一次无效的数据库查询。

建议：添加参数校验：if (StringUtils.isBlank(goodsId)) { return null; }
建议 弹窗无查询结果缓存，重复点击同一商品发起多次请求 dialog_directive.js:3457 性能
问题分类：性能

每次点击商品名称都发起 HTTP POST 请求。虽然 loading 标志位防止了并发重复请求，但用户关闭弹窗后再次点击同一商品仍需重新请求。

建议：添加 goodsInfoCache 对象（key = goodsId），openDialog 中先检查缓存，命中则直接渲染，避免重复请求。
建议 上市日期未按系统参数【商品资料的上市日期是否精确到月日】控制 GoodsBaseController.java:526-533 & goodsBasicDetailModal.html:107-108 业务逻辑
问题分类：数据展示

系统参数 publish.date.precision.day.flag（GoodsParamConstants.PUBLISH_DATE_PRECISION_DAY_FLAG）用于控制商品资料的上市日期显示格式：

  • true 时：显示完整日期（年月日）
  • false 时：仅显示年份

现有商品详情编辑页（goodsDetailForm.jsp、goodsDetailFormMultiModal.jsp、goodsDetail4Order.jsp）均通过 publishDatePrecisionDayFlag 控制输入方式（年份下拉 vs 日期选择器），且 getGoodsDetail 接口（第 562 行）在响应中返回了该参数。

但本次新增的商品基本信息弹窗存在两处遗漏：

  • 后端：queryGoodsInfoByCommonUi 方法（第 526-533 行）未返回 publishDatePrecisionDayFlag
  • 前端模板：goodsBasicDetailModal.html 第 108 行写死 limitTo:10（始终显示年月日），未根据系统参数控制显示格式
<!-- goodsBasicDetailModal.html:108 -- 始终显示完整年月日，未受系统参数控制 --> <span title="{{goodsInfoModalInfo.publishDate | limitTo:10}}"> {{goodsInfoModalInfo.publishDate | limitTo:10}} </span> // GoodsBaseController.java:526-533 -- 未返回 publishDatePrecisionDayFlag public Object queryGoodsInfoByCommonUi(@RequestBody String goodsId) { GoodsDetailValue goodsDetailValue = goodsBaseService.getGoodsBasicInfo(goodsId); goodsDetailValue.setImage(InvUtils.billImageOssUrl(goodsDetailValue.getImage())); Map<String, Object> map = new HashMap<>(); map.put("goodsInfo", goodsDetailValue); return map; // ← 缺少 publishDatePrecisionDayFlag } // 对比 getGoodsDetail（第 562 行）—— 正确返回了该参数： map.put("publishDatePrecisionDayFlag", SysParamsUtils.getBooleanSystemParam( GoodsParamConstants.PUBLISH_DATE_PRECISION_DAY_FLAG, false));
建议：
  • 后端：在 queryGoodsInfoByCommonUi 方法的返回 Map 中添加 publishDatePrecisionDayFlag
  • 前端指令：在 dialog_directive.js 的 openDialog 回调中提取该参数并挂到 scope
  • 前端模板：根据 publishDatePrecisionDayFlag 判断上市日期显示格式 —— true 时显示 limitTo:10（年月日），false 时显示 limitTo:4（仅年份）
